ConwayLife Sprint3

Introduction

Realizzazione in Java del GAME OF LIFE DI CONWAY: Evoluzone del sistema locale utilizzando una GUI basata su web.

Requirements

  • R1: Realizzare una versione in Java del gioco Life di Conway, come gioco zero-player.
    Il gioco consiste nell'introdurre una Griglia di Celle il cui stato (cella 'viva' o cella 'morta') evolve come stabilito dallle regole di ConwayLife
  • R2: L'utente umano deve poter:
    • specificare la configurazione iniziale della griglia del gioco
    • vedere l'evoluzione del gioco in forma opportuna (si veda Problema della vista del gioco )
    • fermare e far ripartire l'evoluzione del gioco
    • pulire (a gioco fermo) la configurazione della griglia del gioco
  • R3: Utilizzare una pagina HTML come dispositivo di I/O per il gioco.
  • R4: La pagina deve costituire un componente esterno alla applicazione Java, si devono poter aprire più pagine HTML per lo stesso gioco, le quali devono essere costantemente aggiornate dal server.
  • R5: l'utente che ha aperto per primo il gioco sarà l'owner, solo la pagina dell'owner avrà pulsanti di comando START/STOP/CLEAN/EXIT attivi.
  • R6: La pagina HTML deve essere aggiornata automaticamente man mano il gioco procede.
  • R7: Un utente non owner che si collega mentre il gioco è in corso, deve vedere lo stato attuale della griglia in modo corretto.
  • R8: Opzionalmente: la pagina HTML deve indicare se il gioco continua anche nel caso di griglia vuota o di configurazione stabile (gioco fermo).
  • R9: Il deployment del gioco avviene mediante Docker.

Requirement analysis

Il dominio principale consiste in:
  • Cella: Un'entita che puo essere viva o morta.
  • Griglia: Una struttura 2D che contiene le celle e ne gestisce lo stato.
  • Regole: La logica che determina lo stato di una cella nella generazione successiva in base ai suoi vicini.
L'interazione richiede:
  • Input: Comandi dall'utente (start, stop, clear, cambio stato cella).
  • Output: Visione dello stato della griglia.
Poiche la GUI è basata su web, il sistema deve gestire la comunicazione remota tra il motore di gioco (Java) e il client (HTML/JS).

Problem analysis

Utilizziamo l'astrazione IOutDev per rappresentare un dispositivo di output. Poiche la GUI è in un browser, abbiamo bisogno di un server che serva file statici e permetta di ricevere aggiornamenti in tempo reale tramite WebSockets (evitando il polling). Il componente server (Javalin) deve agire da ponte tecnologico, il LifeController gestisce tutta logica applicativa.

Test plans

L'utente verifichi che:
  • I comandi inviati dal browser attivino correttamente le azioni nel LifeController.
  • Gli aggiornamenti della griglia nel dominio siano correttamente propagati al client web tramite IOutDev.

Project

L'implementazione si basa su:
  • Javalin: Un framework web leggero per Java.
  • WebSockets: Utilizzati per la comunicazione full-duplex tra il server e il browser.
  • Docker: Il sistema è containerizzato per facilitare il deployment.
Il punto di ingresso del sistema e MainConwayGui, che inizializza la logica di dominio e il ponte server.

Testing

Viene utilizzato JUnit. Si ha la possibilità di automatizzare i test con gradle.

Deployment

La build del progetto avviene tramite gradle. Il deployment del sistema avviene tramite Docker: docker build -t conway-sprint3 . docker compose up -d
